3. Création d’images Docker

Créer une image en utilisant chaque méthode en se basant sur l’image ubuntu:20.04 et contenant le serveur web Nginx utilisant la configuration ci-dessous : 

user www-data;
daemon off;
worker_processes auto;
pid /run/nginx.pid;
include /etc/nginx/modules-enabled/*.conf;
 
events {
    worker_connections 768;
}
 
http {
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    types_hash_max_size 2048;
 
    include /etc/nginx/mime.types;
    default_type application/octet-stream;
 
    ssl_protocols TLSv1 TLSv1.1 TLSv1.2; # Dropping SSLv3, ref: POODLE
    ssl_prefer_server_ciphers on;
 
    access_log /var/log/nginx/access.log;
    error_log /var/log/nginx/error.log;
 
    gzip on;
 
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}

Le serveur web doit se lancer et répondre à l'adresse http://localhost:8080.

Deux méthodes existent pour créer des images Docker.

Manuelle

Lancer un container, exécuter des commandes, puis une fois l’image satisfaisante, utiliser la commande docker commit sur le container pour créer une nouvelle image.

Dockerfile

Créer un fichier Dockerfile scriptant l’ensemble des instructions pour créer une image à partir d’une image de base. Le lancement par défaut de l'image doit automatiquement exécuter le service.

Références :

  • https://docs.docker.com/engine/reference/builder/

  1. Démarrer un container de base de données PostgreSQL utilisant l’image Docker officielle postgres.

  2. Créer une base de données, puis avec l’outil ‘psql’ de l‘image, créer une table et quelques lignes de données à l’aide des commandes SQL suivantes : 

    CREATE TABLE users (id SERIAL PRIMARY KEY, username TEXT);

  3. Arrêter le container, redémarrer le container, vérifier que les données sont encore présentes

  4. Arrêter le container, supprimer le container, redémarrer un container PostgreSQL, vérifier l’état des données. Faire en sorte que les données soient persistantes entre les suppression/recréations de container en utilisant les volumes Docker.

5. Compositions de services

Récupérer le code du dépôt git suivant https://github.com/Soulou/sample-tp-siris-container.

Le but est de faire fonctionner l’ensemble des composants de cette application dans des containers. Pour cela nous allons dans un premier temps construire l’image avec les dépendances du projet à l’aide d’un Dockerfile.

Une fois l’image prête, utiliser l’outil docker-compose pour démarrer l’ensemble des services/containers en une seule commande. Docker Compose est utilisé pour exécuter des applications en développement, pour que tous les membres d’une équipe aient exactement le même environnement.

Références :

  • https://docs.docker.com/compose/

6. Limitation de ressources

Mémoire

Démarrer un container avec une limite de mémoire de 256Mo de RAM, et exécuter l’utilitaire stress au sein de ce container pour allouer 512Mo de RAM. Observer ce qu’il se passe. Depuis la machine hôte, changer la limite à chaud en modifiant les paramètre du cgroup memory, pour le container dans lequel on a executé la commande memory. Augmenter la limite à 1024Mo, ensuite réexécuter la commande dans le container et vérifier que tout se passe bien.

Pour information, il y a aussi une commande docker pour effectuer cette modification, mais le but de cet exercice était de montrer comment manipuler un cgroup à la main.

Références :

  • https://repology.org/project/stress/information
  • https://packages.ubuntu.com/focal/stress
  • https://www.kernel.org/doc/gorman/html/understand/understand016.html
  • https://www.kernel.org/doc/Documentation/cgroup-v1/memory.txt

CPU

Utiliser le cgroup cpuset pour limiter un container à l’usage d’un 1 CPU maximum, lancer la commande stress avec une option la faisant utiliser 2 CPUs puis observer le comportement (par ex. avec la commande top ou htop)

Démarrer deux containers utilisant un seul CPU, avec des priorités (poids) différents 512 et 1024 par exemple. Exécuter la commande stress comme précédemment et constater le résultat. Le container le plus prioritaire devrait être capable d’utiliser deux fois plus de temps CPU que l’autre (66% vs 33%)
